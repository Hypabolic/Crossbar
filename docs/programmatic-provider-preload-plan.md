# Programmatic provider preload plan

## Objective

Register saved Crossbar providers before Pi performs initial model resolution.

The completed change must:

- remove Pi's misleading startup `No models available` warning when Crossbar has usable cached models;
- make saved Crossbar models available to interactive startup, `--list-models`, print mode, JSON/RPC mode, model scopes, and CLI model selection;
- continue to work offline from the persisted model catalogue;
- keep network discovery, health polling, widgets, and other session-scoped work out of the extension factory.

This is an implementation handoff. Do not redesign the recently changed `/crossbar` UI while doing this work.

## Confirmed root cause

Crossbar currently registers saved providers inside `session_start` in `src/index.ts`.

Pi 0.79.9 performs the relevant operations in this order:

1. Create `AuthStorage`, `ModelRegistry`, and the resource loader.
2. Load and await extension factories.
3. Flush provider registrations queued by `pi.registerProvider(...)` during factory execution into `ModelRegistry`.
4. Resolve model scopes and the initial/CLI model.
5. Handle `--list-models`, print, JSON, RPC, or interactive startup.
6. Create a session and emit `session_start` where applicable.

Therefore registration in `session_start` is too late for:

- the initial interactive model;
- `pi --list-models`;
- `pi --provider ... --model ...`;
- provider-prefixed `--model provider/model`;
- print/JSON/RPC modes that need a model before session events;
- offline programmatic startup.

Pi explicitly recommends an async extension factory for dynamic provider registration:

- `node_modules/@earendil-works/pi-coding-agent/docs/extensions.md`
- `node_modules/@earendil-works/pi-coding-agent/docs/custom-provider.md`

Factory-time `pi.registerProvider(...)` calls are queued and flushed before model resolution.

## Architectural decision

Split Crossbar startup into two phases.

### Phase 1: factory preload

The extension factory becomes async and performs only durable-cache registration:

```ts
export default async function crossbar(pi: ExtensionAPI): Promise<void> {
  await preloadCachedProviders(pi);
  // Register events and commands as today.
}
```

`preloadCachedProviders` must:

1. Read `crossbar.json` using the existing tolerant persistence API.
2. Select enabled records with at least one cached non-embedding model.
3. Build provider configuration using the same pure `buildProviderConfig(...)` used by live registration.
4. Call `pi.registerProvider(record.id, config)`.
5. Never perform network requests, discovery, polling, UI work, timers, or credential writes.
6. Never throw out of the extension factory. A malformed record must not prevent Pi from starting.

The factory does not construct a `ServerRegistry` and does not use `registerServer(...)`, because live `registerServer` validates credentials through the registry.

### Phase 2: session startup

Keep `session_start` responsible for:

- constructing the `ServerRegistry` backed by Pi's `AuthStorage`;
- refreshing models from reachable servers;
- auto-discovery;
- health polling;
- the loaded-model status widget;
- live re-registration when the model catalogue changes;
- user notifications.

Factory registration and later registration use the same stable provider ID. Pi's `ModelRegistry.registerProvider` replaces the existing provider model list; it does not append a duplicate provider.

## Authentication rules

Use the current provider-config behavior unchanged:

- `auth: "none"` → literal `crossbar-no-auth`;
- `auth: "apiKey"` → `$CROSSBAR_...` sentinel generated by `envVarFor(record.id)`.

Factory preload must not call `registerServer(...)` and must not reject a keyed provider merely because the factory cannot access an `ExtensionContext`.

Register the keyed provider with its sentinel. Pi's already-created `AuthStorage` is consulted by `ModelRegistry`:

- stored key present → model is available;
- key absent → model remains registered but is excluded from `getAvailable()` and fails through Pi's normal missing-auth path;
- missing auth must not abort extension loading or affect unrelated providers.

Live management operations may continue to give a more specific Crossbar error when a keyed server has no stored credential.

## Persisted model catalogue

Factory preload depends on `ServerRecord.lastKnownModels`. It is not reliable enough currently.

`ServerRegistry.updateHealthCache(...)` intentionally does not persist. Both startup refresh and polling currently update this ephemeral cache without guaranteeing that a new Pi process sees the updated catalogue.

Add an explicit persistent operation, for example:

```ts
async setLastKnownModels(id: string, models: ModelDescriptor[]): Promise<void>
```

Required behavior:

- update the record's `lastKnownModels`;
- persist through the registry's existing atomic `flush()` path;
- write only when the catalogue materially changed;
- do not persist health state or `lastSeenAt` on every polling interval.

Use one consistent comparison rule. At minimum compare model IDs order-independently, matching `modelsChanged(...)`. Prefer including registration-relevant metadata if changes to context window, modalities, reasoning, tools, or output limit must survive restart.

Decide and document the comparison contract before implementation. ID-only comparison is insufficient if a changed context window must be preloaded on the next process.

Update these paths:

1. Manual add: already stores the fetched list in the new record; retain this.
2. Auto-discovery: ensure the fetched models are persisted after first registration.
3. Startup live refresh: persist successful changed catalogues.
4. Polling: persist only inside the changed-catalogue branch.
5. Any switch/load flow that refreshes registration-relevant model metadata should use the same persistent update helper.

`saveConfig(...)` already writes atomically using a sibling temporary file and rename. Preserve that implementation.

## Suggested code structure

### `src/shim/provider-shim.ts`

Add or export a cache-only helper if useful:

```ts
export function registerCachedServer(
  pi: ExtensionAPI,
  record: ServerRecord,
  models: ModelDescriptor[],
): boolean
```

Possible behavior:

- filter chat models;
- return `false` without registering when none remain;
- construct the minimal `DiscoveredServer`;
- call `buildProviderConfig`;
- call `pi.registerProvider`;
- return `true`.

Alternatively keep this logic in `src/index.ts`, but there must be only one provider-config mapping implementation.

Do not weaken live keyed-credential validation merely to support preload. Keep preload and live validation separate.

### `src/index.ts`

1. Change the default export to async.
2. Load persisted config before installing lifecycle handlers.
3. Preload enabled cached providers.
4. Reuse the loaded configuration as initial factory state only if safe; `session_start` must still reload from disk because reload/new/resume creates a fresh extension instance and external state may have changed.
5. Leave event/command registration in the factory.
6. Do not start timers until `session_start`.

### `src/registry/registry.ts`

Add a persistent catalogue update method. Keep `updateHealthCache` ephemeral for health/widget updates.

### `src/poll.ts`

When a successful `listModels` result differs:

1. persist the changed catalogue;
2. re-register the provider;
3. avoid duplicate persistence after the same change.

Carefully choose ordering:

- persisting first ensures a process crash still leaves a valid catalogue;
- registering first avoids persisting a catalogue that fails provider validation.

Recommended: build/validate config, persist, then register. Ensure errors leave the previous registration/cache intact where possible.

## Reload and session behavior

Test these lifecycle cases explicitly:

- initial process startup;
- `/reload`;
- new session;
- resume;
- fork;
- session shutdown.

Each new extension instance runs the factory and preloads from cache. `session_start` then refreshes live state.

Provider registration is keyed by `record.id`, so repeated registration must replace the same provider. Confirm no temporary unregister occurs before the replacement during startup.

Current `session_shutdown` unregisters Crossbar providers. Ensure this does not create a gap during reload/new-session transitions that causes the replacement runtime to lose its factory-preloaded providers. If Pi constructs the replacement services before old shutdown, inspect ordering and add an integration test. Do not remove cleanup without understanding cross-runtime registration ownership.

## Programmatic acceptance criteria

Given an enabled no-auth server with a persisted chat model, all of these must work without an interactive `/crossbar` command:

```bash
pi --list-models crossbar

pi \
  --provider crossbar-llamacpp-10-0-1-252-8088 \
  --model 'model-id' \
  --no-session \
  -p 'Reply with OK'

pi \
  --model 'crossbar-llamacpp-10-0-1-252-8088/model-id' \
  --no-session \
  -p 'Reply with OK'

pi --mode rpc

pi --offline --list-models crossbar
```

Expected behavior:

- cached Crossbar models are present before model resolution;
- no stock `No models available` warning when at least one cached provider is usable;
- no-auth requests resolve with `crossbar-no-auth`;
- keyed providers are available only when Pi has matching auth;
- no factory network request is made;
- offline listing and selection use cached metadata.

Note: a brand-new server with no cached model catalogue cannot be selected programmatically until an interactive discovery/add or another supported setup process persists its catalogue. This is an inherent cache-first limitation and should be documented.

## Test plan

### Unit: factory preload

Add tests for an extracted preload function:

- enabled cached no-auth server registers;
- disabled cached server does not register;
- enabled server without cache does not register;
- embedding-only cache does not register;
- mixed cache registers only chat models;
- multiple enabled servers register independently;
- one malformed record does not block valid records;
- missing/corrupt config results in no registrations and no throw;
- exact placeholder/sentinel values match live registration;
- no discovery, probe, fetch, timer, or UI function is called.

### Unit: keyed availability

Using Pi's in-memory `AuthStorage` and `ModelRegistry`:

- keyed cached provider plus stored credential is available;
- keyed cached provider without stored credential is registered but unavailable;
- missing keyed auth does not affect a no-auth provider;
- no-auth cached provider resolves `crossbar-no-auth`.

### Unit: persistent catalogue

- changed catalogue updates `lastKnownModels` and persists once;
- unchanged catalogue does not persist;
- registration-relevant metadata changes persist according to the chosen comparison contract;
- persistence round-trips through `crossbar.json`;
- embedding filtering/normalization is consistent between persistence and registration;
- concurrent reads never observe partial JSON.

### Lifecycle

- factory registration occurs before `session_start`;
- unchanged live refresh does not unnecessarily unregister/re-register;
- changed live refresh persists and replaces the cached registration;
- reload/new/resume/fork produce one provider per stable ID;
- shutdown cleanup does not remove the replacement runtime's registration.

### CLI integration

Use a temporary agent directory and the local Pi executable:

1. Create temporary `settings.json` loading this local extension.
2. Create `crossbar.json` with an enabled cached server.
3. Create `auth.json` when testing keyed providers.
4. Set `PI_CODING_AGENT_DIR` to the temporary directory.
5. Spawn Pi and assert:

   - `--list-models` includes the cached model;
   - `--offline --list-models` includes it;
   - provider/model CLI selection resolves before session startup;
   - print mode does not emit `No models available`;
   - keyed-without-key is unavailable without crashing;
   - RPC startup exposes/accepts the cached model as expected.

Avoid using the user's real `~/.pi` in tests.

## Additional follow-up items

These are not blockers for pre-start registration, but were identified during review:

- vLLM and llama.cpp currently cannot reliably classify embedding-only servers;
- generic ID-only embedding heuristics remain incomplete by nature;
- polling can race a management overlay's switch-and-select sequence;
- manual-add and enable-rollback UI branches need additional tests;
- the loaded-model expiry text is a snapshot, not a ticking countdown.

Do not expand this task into solving all follow-ups unless needed for correctness.
